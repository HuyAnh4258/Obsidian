> [!note]+ # Giới thiệu chung
> > `Object - Oriented Programming` (hay Lập trình hướng Đối tượng, lập trình tập trung vào Đối tượng) là một cách lập trình dựa trên khái niệm về các `Objects` (vật thể), trong đó lưu trữ các thông tin dữ liệu và phương thức hành vi.
> 
> Mục đích chính của `OOP` chính là tính `linh hoạt cao` và `dễ bảo trì` phần mềm. Các tính năng chính của `OOP` bao gồm 4 tính năng:
> > [!note]+ ### Encapsulation (Tính đóng gói)
> > > `Encapsulation` (hay tính đóng gói) là một cơ chế `đóng gói dữ liệu` (và các phương thức) trong một `Class` để giấu đi việc triển khai cụ thể bên trong khỏi các truy cập bên ngoài.
> > 
> > Nói ngắn gọn, cơ chế này giúp tạo ra `trừu tượng dữ liệu` và `kiểm soát dữ liệu đối với các Class khác` bằng cách sử dụng `kiểu truy cập` (access modifier)
> > [!note]+ ### Inheritance (Kế thừa)
> > > `Inheritance` (sự kế thừa) cho phép các `Class mới` `kế thừa` những đặc điểm sẵn có của một `Class đã có` mà không phải triển khai lại.
> > 
> > Các class con sẽ kế thừa các `attributes` (thuộc tính) và `methods` (phương thức) của class gốc và thêm thắt các đặc điểm độc nhất khác.
> > > [!note] 🗣
> > > Java chỉ hỗ trợ `kế thừa đơn`, KHÔNG hỗ trợ `đa kế thừa` nhưng lại hỗ trợ `kế thừa đa tầng` .
> > > → Có nghĩa là:
> > > - Một lớp chỉ có thể kế thừa từ một cha duy nhất
> > > - Nhiều lớp có thể cùng kế thừa từ một cha
> > > - Ta có thể kế thừa đa tầng như `ông - cha - con` tạo thành một chuỗi kế thừa (như một cây phả hệ)
> > [!note]+ ### Polymorphism (Đa hình)
> > > `Polymorphism`** (Tính đa hình)** là khả năng cho phép **một đối tượng hoặc phương thức** mang nhiều hình dạng khác nhau. Các **class cha** định nghĩa các hành vi chung (baseline), còn **các class con** triển khai hoặc mở rộng các hành vi đó theo cách riêng.
> > 
> > **Ví dụ dễ hiểu:**
> > Nếu như một việc có thể làm bằng nhiều cách khác nhau, đó chính là tính đa hình. **VD:** Để kêu lên thì các loài động vật có nhiều cách thức và tiếng kêu khác nhau, như con mèo thì kêu meo meo còn con chó thì kêu gâu gâu.
> > Trong Java, điều này thường được thể hiện thông qua **kế thừa (inheritance)** và **ghi đè phương thức (method overriding)**
> > [!note]+ ### Abstraction (Trừu tượng)
> > > `Abstraction` (hay Trừu tượng hoá) là quá trình **ẩn đi các chi tiết không cần thiết** và **chỉ thể hiện những đặc điểm, hành vi cốt lõi** của đối tượng, giúp ta tập trung vào *“cái gì đối tượng làm”* thay vì *“làm như thế nào”*.
> > 
> > → `Abstraction` là quá trình `loại bỏ chi tiết cụ thể`, giữ lại những `đặc điểm tổng quát` và cốt lõi để biểu diễn `nhóm đối tượng có liên quan`, giúp `giảm sự phức tạp` và `tập trung vào bản chất` thay vì cách thực thi.
> > Trong Java, abstraction được thực hiện thông qua `abstract class` và `interface`, tạo nền tảng cho việc mở rộng và triển khai chi tiết ở các lớp con.
> [!note]+ # Encapsulation (Tính đóng gói)
> > [!note]+ ### Định nghĩa
> > > `Encapsulation` (Tính đóng gói) là một định nghĩa `quan trọng` trong `OOP` giúp đúc kết các dữ liệu và phương thức  vào trong một đơn vị duy nhất là `Class` trong Java.
> > 
> > - Bản chất đóng gói là khi ta để `dữ liệu` ở trạng thái `private`
> > - Ta sẽ sử dụng các `phương thức` ở trạng thái `public` để truy xuất đến `dữ liệu`.
> >     - **Tính bảo mật: **Điều này tránh được dữ liệu của ta có thể được truy cập ngẫu nhiên từ các `Class` ngoài hoặc các code ở nơi khác.
> >     - **Dễ bảo trì:** khi ta sửa đổi code sẽ `không gây ảnh hưởng` đến phần code của người khác trên cùng một dự án.
> > 
> > > [!note]+ Hai bước để triển khai `Đóng gói` :
> > > - Để các `biến instance` sang `private` để chúng không thể được `truy xuất` trực tiếp từ các `Class ngoài` , chỉ có thể lấy và sửa đổi các biến này thông qua các `phương thức public` của `Class đó` .
> > > - Triển khai các `getter` và `setter` để tái định nghĩa và truy xuất dữ liệu.
> > 
> > > [!note]+ Lợi ích của `đóng gói`:
> > > - **Ngăn ngừa chỉnh sửa vô ý:** các dữ liệu được đóng gói sẽ được `bảo vệ` khỏi các thay đổi từ các `Class khác`
> > > - **Áp đặt chuẩn hoá dữ liệu:** chỉ có các phương thức có `thẩm quyền` mới có thể thao túng được dữ liệu → đảm bảo được tính `đúng đắn` và `nhất quán`.
> > > - **Tăng bảo mật:** `bảo vệ` các thông tin nhạy cảm khỏi các truy cập và sửa đổi không được phép.
> > [!note]+ ### Getter và Setter
> > ```java
> > public class MyClass {
> > 	private String data; //data is private - other class cant access
> > 	
> > 	public String getData(){
> > 		return this.data; //Use this method to get the data outside
> > 	}
> > 	
> > 	public void setData(String data){
> > 		this.data = data; //Use this method to set the value from outside
> > 	}
> > }
> > ```
> > Với `getter` và `setter` , ta có thể chuẩn hoá được dữ liệu và kiểm soát được việc truy xuất đến dữ liệu đó:
> > ```java
> > public void setAge(int age){
> > 	if (age < 0 || age > 100){
> > 		throw new IllegalArgumentException();
> > 	}
> > 	this.age = age;
> > }
> > ```
> > Naming Convention: cách đặt tên chuẩn cho `Getter` và `Setter`
> > - Getters:
> >     - Luôn luôn bắt đầu bằng từ khoá `get`
> >     - Viết hoa chữ cái đầu của thuộc tính (theo `camelCase`)
> >     - Đặc biệt với `boolean` thì ta dùng từ khoá `is` thay vì `get` đứng đầu.
> > - Setters:
> >     - Luôn luôn bắt đầu bằng từ khoá `set`
> >     - Viết hoa chữ cái đầu của thuộc tính (theo `camelCase`)
> >     - Đặt đối số `argument` tương ứng với dữ liệu tham số truyền vào.
> > [!note]+ ### Java Package (Gói)
> > Ngoài ra, tính đóng gói còn thể hiện với `Java Package`
> > > `Java Package` là một `gói` để gộp các `Class` , `Interface` hoặc các `sub-package` tương đồng. 
> > 
> > Có 2 loại package:
> > - `Built-in` : Các package có sẵn để chứa các đoạn mã thư viện của Java.
> > - `User-defined`: Do người dùng tự định nghĩa để phân bổ code một cách khoa học.
> > 
> > Có 3 cách truy cập:
> > ```java
> > import package_name.ClassName; //Khai báo chi tiết class của package cần dùng
> > import package_name.*; //Khai báo toàn bộ gói cha
> > package_name.ClassName object = new package_name.ClassName();
> > //Truy cập bằng tên đầy đủ (fully qualified name)
> > ```
> > Ví dụ
> > ```java
> > import java.util.Scanner; //c1
> > import java.util.*; //c2
> > 
> > java.util.Scanner sc = new java.util.Scanner(System.in); //c3
> > ```
> > > [!note] 🗣
> > > Lưu ý rằng mỗi một file `.java` chỉ có `duy nhất` một câu lệnh `package` và nó phải nằm ở `dòng code đầu tiên`.
> > [!note]+ ### Access Modifier (Phạm vi truy cập)
> > > `**Access Modifier**` (bộ truy cập / phạm vi truy cập) là `**từ khóa**` dùng để **quy định phạm vi **`**truy cập**`** (mức độ nhìn thấy)** của `**class, method, variable, hoặc constructor**` trong Java.
> > 
> > > [!note] 🗣
> > > **Nói đơn giản:** kiểm soát `“ai được dùng cái gì”` trong chương trình
> > 
> > Có 4 loại:
> > - `private`: Chỉ truy cập được bên trong cùng `Class` đó.
> > - `default`: Chỉ truy cập được bên trong cùng `Package`, các class trong cùng package có thể dùng chung. Đây là kiểu `mặc định` nếu như ta không chỉ định.
> > - `protected`: Chỉ truy cập được bên trong cùng `package` và các `Class con` bên ngoài package đã kế thừa `Class` chứa nó hoặc chính class đó.
> > - `public`: Có thể được truy cập ở tất cả mọi nơi trong cùng `project`.
> > 
> > Các từ khoá khác: static, abstract, synchronized, native, volatile, transient, etc.
> > [!note]+ ### `this` Keyword
> > > Từ khoá `this` được dùng để chỉ đến `chính đối tượng hiện tại` khi ta dùng bên trong một `method` hoặc `constructor` của một `Class`.
> > 
> > ```java
> > public class Point {
> > 	private int x;
> > 	private int y;
> > 	
> > 	public void setX(int x) {
> > 		this.x = x; // Avoiding confusion with parameter
> > 	}
> > 	
> > 	public int getX() {
> > 		return this.x; // Accessing object member
> > 	}
> > }
> > ```
> > Ta cũng có thể dùng `this` để gọi ra `constructor` của một `Class`
> > ```java
> > public class Account {
> > 	private String name;
> > 	private double balance;
> > 	
> > 	public Account(String name) {
> > 		this(name, 0.0); // Calling another constructor
> > 	}
> > 	
> > 	public Account(String name, double balance) {
> > 		this.name = name;
> > 		this.balance = balance;
> > 	}
> > }
> > ```
> [!note]+ # Inheritance (Tính kế thừa)
> > [!note]+ ### Định nghĩa
> > > `Inheritance` cho phép ta tạo ra các `Class mới` dựa trên `Class cũ` bằng cách kế thừa và mở rộng từ một class có sẵn. 
> > 
> > Điều này giúp `tái sử dụng code` bằng cách:
> > - Kế thừa toàn bộ dữ liệu và hành vi
> > - Mở rộng thêm các dữ liệu hoặc hành vi mới.
> > > Sự kế thừa đại diện cho quan hệ `is a` trong quan hệ thực thể. (`cha - con`)
> > [!note]+ ### Các khái niệm chính
> > - `Superclass` (Lớp cha): là `Class có sẵn` mà các class khác sẽ kế thừa
> > - `Subclass` (Lớp con): là `Class mới` kế thừa các `đặc điểm` và `hành vi` của lớp cha.
> > - `Inheritance Hierarchy`(Phân cấp kế thừa): là quan hệ giữa `Cha` và `các Con` , tạo thành cấu trúc cây.
> > 
> > 
> > [!note]+ ### Các loại kế thừa
> > ![[image 17.png]]
> > [!note]+ ### `extends` keyword
> > Từ khoá `extends` dùng để tạo sự kế thừa giữa hai `Class`
> > ```java
> > class Super {
> > .....
> > }
> > 
> > class Sub extends Super {
> > .....
> > }
> > ```
> > 
> > [!note]+ ### Một số thuật ngữ về kế thừa
> > ![[image 18.png]]
> > - `is a` là quan hệ dựa trên kế thừa, ta coi `Class con` như một kiểu của `Class cha`
> > - `has a` là quan hệ bao gồm, một `Class` bao gồm một `Class` khác như một thuộc tính.
> > ```java
> > class Engine{
> > 	int velocity;
> > 	boolean isFucked;
> > }
> > 
> > class Car {
> > 	String brand;
> > 	double price;
> > 	Engine engine; //Car has an engine to run
> > }
> > ```
> > - Kế thừa `trực tiếp` và `gián tiếp`:
> > ![[image 19.png]]
> > Khi class `con` kế thừa một class `cha` cũng đang kế thừa một class `ông` 
> > > [!note] 🗣
> > > Ta nói: 
> > > - `con` kế thừa trực tiếp `cha` (kế thừa 1 tầng)
> > > - `con` kế thừa gián tiếp `ông` (kế thừa 2 tầng hoặc hơn)
> > 
> > 
> > [!note]+ ### `super` keyword
> > > `super` là từ khoá dùng để truy cập đến các `dữ liệu và phương thức` (không private) của lớp cha.
> > 
> > ![[image 20.png]]
> > Ta dùng `super()` để gọi đến `constructor` của lớp cha, ta có thể `truyền tham số` vào dựa trên constructor ta muốn dùng.
> > ![[image 21.png]]
> > Từ khoá `super` chỉ tham chiếu được đến `class cha gần nhất`, không thể tham chiếu xa hơn. Ta bắt buộc phải `thông qua method` của class gần nhất để tham chiếu đến class cha trên đó.
> > > [!note] 🗣
> > > **KHÔNG THỂ** sử dụng `super.super()`
> > 
> > 
> > [!note]+ ### `final` keyword
> > > Một `Class` được đặt keyword `final` sẽ không thể được kế thừa bởi  bất kỳ class nào khác.
> > 
> > > [!note] 🗣
> > > Dễ hiểu hơn:
> > > - `final class` là lớp cuối cùng của nhánh đó, `không có hậu duệ` (kế thừa)
> > > - Vì không thể kế thừa từ `final class` , ta không có lý do gì và cũng không thể dùng `final` cho `abstract class` (cốt lõi là để tạo base class cho các class khác mở rộng)
> > [!note]+ ### Constructor and Finalizers (Hàm khởi tạo & Hàm huỷ)
> > > `Constructor` là một `method đặc biệt` được gọi tự động khi ta dùng `new` keyword để tạo ra một đối tượng mới. Nó dùng để `khởi tạo giá trị` ban đầu cho đối tượng.
> > 
> > - Việc gọi các `constructor khác` như `this()` hay `super()` để tạo nên một `constructor` được gọi là `constructor chaining`
> > 
> > > `Finalizer` là một phương thức đặc biệt được gọi khi đối tượng bị `GC` đánh dấu và chuẩn bị xoá khỏi bộ nhớ. Dùng để giải phóng tài nguyên.
> > 
> > ![[image 22.png]]
> > - Đây là một method thuộc `java.lang.Object`
> > - Được gọi tự động bởi GC - không đảm bảo thực hiện ngay
> > - Java không đảm bảo phương thức này được gọi trước khi chương trình kết thúc
> > 
> > > [!note] 🗣
> > > Từ Java 9 thì `finalize()` đã bị bỏ, thay vào đó ta dùng `try-with-resources()` để đóng tài nguyên tự động sau khi thực thi một khối lệnh liên quan đến tài nguyên (File, Network, DB Connection,…)
> > [!note]+ ### Casting Objects (Đổi kiểu Đối tượng)
> > > Java cho phép một `Class con` có thể được coi như một`Đối tượng Class cha` của nó. Đây gọi là `upcasting` (biến đổi lên). 
> > 
> > ```java
> > Parent p = new Child(); //this is upcasting (automatically)
> > 
> > Child c = new Parent(); //Compile error (need to be done manually)
> > 
> > Child c = (Child) new Parent(); //Downcasting: throws ClassCastException
> > ```
> > > [!note]+ ### `instanceof` Operator
> > > > `instanceof` là toán tử để kiểm tra xem một `đối tượng` có phải là một `phiên bản` của một `Class` hoặc một `Interface` hay không. Nó trả về `true` hoặc `false` dựa trên việc so sánh kiểu.
> > > 
> > > ```java
> > > class Animal{}
> > > class Dog extends Animal{//Dog inherits Animal
> > > 	public static void main(String args[]){
> > > 		Animal a = new Animal();
> > > 		Dog d = new Dog();
> > > 		System.out.println(a instanceof Animal);//true
> > > 		System.out.println(d instanceof Animal);//true
> > > 	}
> > > }
> > > ```
> > > 
> > > [!note]+ ### 
> > > ```java
> > > class Animal {
> > > }
> > > class Dog extends Animal {
> > > }
> > > public class Main {
> > > 	public static void main(String[] args) {
> > > 		Animal animal = new Dog();
> > > 		Dog d = (Dog) animal;
> > > 		System.out.println("Downcasting performed");
> > > 	}
> > > }
> > > ```
> [!note]+ # Polymorphism (Tính đa hình)
> > [!note]+ ### Overview
> > > Tính đa hình (sự đa hình thái) cho phép các object của các Class khác nhau thực hiện hành vi khác nhau `khi ta gọi chung một phương thức`. Tức là hai Class này cùng chia sẻ một kiểu hành vi, nhưng lại thực hiện khác nhau.
> > 
> > **Ví dụ:**
> > - Với `Động Vật`, ta có nhiều loài động vật khác nhau như `Chó`, `Mèo`, `Chim`.
> > - `Động Vật` chia sẻ nhiều thông tin như tên, tuổi hay hành vi như ăn để sống, ngủ, kêu,…
> > - Vì vậy chúng ta sẽ sử dụng tính đa hình để mô tả đặc điểm chung này của chúng.
> > ```java
> > public class Animal {
> > 	String name;
> > 	int age;
> > 	
> > 	public void eat(){
> > 		System.out.println(this.name + " is eating!")
> > 	}
> > 	
> > 	public void sleep(){
> > 		System.out.println(this.name + " is sleeping!")
> > 	}
> > 	
> > 	public void makeSound(){
> > 		System.out.println(this.name + " is making sound!")
> > 	}
> > }
> > ```
> > - Từ ví dụ trên, ta sẽ tạo ra các Class động vật phân loại khác như chó, mèo,…
> > ```java
> > public class Cat extends Animal {
> > 	public Cat(String name, int age){
> > 		super(name, age); //Create a Cat as an Animal
> > 	}
> > 	
> > 	//Overriding - behaviour changes denpends on type of animal
> > 	@Override
> > 	public void eat(){
> > 		System.out.println(this.name + " is eating a Fish!")
> > 	}
> > 	
> > 	@Override
> > 	public void sleep(){
> > 		System.out.println(this.name + " is sleeping on the couch!")
> > 	}
> > 	
> > 	@Override
> > 	public void makeSound(){
> > 		System.out.println(this.name + " is making Meow Meow sound!")
> > 	}
> > }
> > ```
> > [!note]+ ### Types
> > > [!note]+ Static Polymorphism
> > > > `Đa hình tĩnh` - hay còn gọi là `Đa hình khi biên dịch` hoặc `xác định sớm` .
> > > Nó cho phép một tên hàm hoặc toán tử có thể có `nhiều cách thực hiện` khác nhau, tùy thuộc vào `kiểu dữ liệu` hoặc số lượng `tham số` truyền vào.
> > > 
> > > - **Cách thực hiện:**
> > >     - `Nạp chồng hàm` (`Function Overloading`): nhiều hàm cùng tên nhưng khác kiểu hoặc số lượng tham số.
> > >     - `Nạp chồng toán tử` (`Operator Overloading`): định nghĩa lại cách hoạt động của các toán tử cho kiểu dữ liệu do người dùng định nghĩa (như trong C++).
> > > 
> > > ```java
> > > class Calculator {
> > > public:
> > >     int add(int a, int b) { 
> > > 	    return a + b; 
> > > 	    }
> > > 	    
> > >     double add(double a, double b) { 
> > > 	    return a + b; 
> > > 	    }
> > > };
> > > ```
> > > > [!note]+ Operator overloading
> > > > > `Operator Overloading` là quá trình nạp chồng các toán tử để thực hiện các chức năng khác nhau. Tức là ta sẽ khiến cho các phép toán +, -, *,… có chức năng khác ngoài thực hiện phép tính với kiểu số.
> > > > 
> > > > Trong Java, trường hợp duy nhất mà Java hỗ trợ nạp chồng thêm chức năng ở đây chỉ có toán tử `+` , có thể được sử dụng để cộng hai số.
> > > > ```java
> > > > int a = 10;
> > > > int b = 20;
> > > > int c = a + b;
> > > > ```
> > > > Nhưng với nạp chồng toán tử, ta có thể lấy logic cộng này để cộng chuỗi `concatenation`
> > > > ```java
> > > > String str1 = "Hello"
> > > > String str2 = "World"
> > > > String result = str1 + str2
> > > > ```
> > > 
> > > > [!note]+ Method Overloading
> > > > > Nạp chồng phương thức `Method Overloading` là kiểu thay đổi đầu vào và ra của một hàm để tạo ra một hàm mới đưa ra các kết quả dạng khác, mặc dù chức năng logic của chúng giống nhau.
> > > > 
> > > > ```java
> > > > public class Shapes {
> > > > 	public void area() {
> > > > 	System.out.println("Find area ");
> > > > 	}
> > > > 	public void area(int r) {
> > > > 	System.out.println("Circle area = " + 3.14 * r * r);
> > > > 	}
> > > > 	public void area(double b, double h) {
> > > > 	System.out.println("Triangle area=" + 0.5 * b * h);
> > > > 	}
> > > > 	public void area(int l, int b) {
> > > > 	System.out.println("Rectangle area=" + l * b);
> > > > 	}
> > > > }
> > > > ```
> > > > Nạp chồng hàm có các đặc điểm sau:
> > > > - <u>**Bắt buộc**</u> phải thay đổi đối số nhận vào, không được trùng lặp
> > > > - Có thể thay đổi kiểu dữ liệu trả về `return type`
> > > > - Có thể thay đổi kiểu truy cập `access modifier`
> > > > - Nạp chồng chỉ xảy ra ở cùng một `Class` hoặc các `subclass` (lớp con) của nó
> > > > 
> > > > > [!note] 🗣
> > > > > **Lưu ý:**
> > > > > - Các hàm chung tên, khác đối số mới được coi là nạp chồng
> > > > > - Nếu các hàm có chung tên, chung đối số nhưng khác kiểu chả về cũng không được coi là nạp chồng → Compiler báo lỗi ngay lập tức
> > > 
> > > > [!note]+ Type Promotion
> > > > > `Type Promotion` hay còn gọi là `Nâng kiểu dữ liệu` xảy ra khi ta chuyển giá trị từ một kiểu dữ liệu nhỏ hơn thành một kiểu dữ liệu lớn hơn.
> > > > 
> > > > ```java
> > > > byte b = 10;
> > > > int i = b;  // byte được "promote" lên int
> > > > ```
> > > > Java sẽ thực hiện `Implicit Conversion` (biến đổi ngầm định) như sau:
> > > > ```java
> > > > byte → short → int → long → float → double
> > > > char → int → long → float → double
> > > > ```
> > > > 
> > > > Trong `Method Overloading` , `Type Promotion` diễn ra như sau:
> > > > - Khi bạn **nạp chồng phương thức**, Java chọn phương thức **phù hợp nhất** dựa trên:
> > > >     1. Kiểu dữ liệu chính xác (exact match) của đối số.
> > > >     2. Nếu không có kiểu chính xác, Java sẽ **thực hiện type promotion** để tìm phương thức phù hợp hơn.
> > > > 
> > > > ```java
> > > > class Demo {
> > > >     void show(int a) {
> > > >         System.out.println("int version");
> > > >     }
> > > > 
> > > >     void show(double a) {
> > > >         System.out.println("double version");
> > > >     }
> > > > 
> > > >     public static void main(String[] args) {
> > > >         Demo d = new Demo();
> > > >         d.show(5);      // (1)
> > > >         d.show(5.5f);   // (2)
> > > >         d.show('A');    // (3)
> > > >     }
> > > > }
> > > > ```
> > > 
> > > > [!note]+ Shadowing
> > > > > `Shadowing` (sự che khuất) xảy ra khi một biến (hoặc static method) trong một `subclass` có cùng tên với một biến (hoặc static method) ở trong `parent` . Khi đó, thành phần trong `subclass` sẽ che mất thành phần trong `parent` nhưng `KHÔNG GHI ĐÈ` 
> > > > 
> > > > ```java
> > > > class Parent {
> > > >     String name = "Parent";
> > > > }
> > > > 
> > > > class Child extends Parent {
> > > >     String name = "Child"; // shadowing field
> > > > }
> > > > 
> > > > public class Test {
> > > >     public static void main(String[] args) {
> > > >         Parent p = new Child();
> > > >         System.out.println(p.name); //Parent
> > > >         Child c = new Child();
> > > >         System.out.println(c.name); //Child
> > > >     }
> > > > }
> > > > ```
> > > > > Điều này tương tự với các static method — vốn thuộc về class chứ không phải object. Vì vậy, chúng không thể override mà chỉ có thể bị che khuất (hidden) khi lớp con định nghĩa lại cùng tên. Đây là `method hiding`
> > > > 
> > > > ```java
> > > > class Parent {
> > > >     static void greet() {
> > > >         System.out.println("Hello from Parent");
> > > >     }
> > > > }
> > > > 
> > > > class Child extends Parent {
> > > >     static void greet() { // static method hiding
> > > >         System.out.println("Hello from Child");
> > > >     }
> > > > }
> > > > 
> > > > public class Test {
> > > >     public static void main(String[] args) {
> > > >         Parent p = new Child();
> > > >         p.greet();   // Hello from Parent
> > > >         Child.greet(); // Hello from Child
> > > >     }
> > > > }
> > > > ```
> > > > > [!note] 🗣
> > > > > **Tóm lại: **`**static methods**`** và **`**fields**` đều:
> > > > > - Thuộc **class**, không phải **instance (object)**
> > > > > - **Không tham gia overriding**
> > > > > - Khi subclass định nghĩa lại → **chỉ che khuất (shadow/hide)** bản gốc của class cha
> > > > > - JVM quyết định phương thức nào được gọi **tại compile-time**, dựa trên **reference type**
> > > 
> > > > [!note]+ Method hiding
> > > > > `Method hiding` xảy ra khi một phương thức `static` trong lớp con có cùng tên, cùng tham số, cùng kiểu trả về với một phương thức `static` trong lớp cha. 
> > > > 
> > > > Vậy khi ta gọi method này với kiểu của lớp con, Java sẽ bind đến method của lớp con (lớp cha bị che khuất)
> > > > ```java
> > > > class Parent {
> > > >     static void display() {
> > > >         System.out.println("Parent static method");
> > > >     }
> > > > }
> > > > 
> > > > class Child extends Parent {
> > > >     static void display() {  // method hiding
> > > >         System.out.println("Child static method");
> > > >     }
> > > > }
> > > > 
> > > > public class Main {
> > > >     public static void main(String[] args) {
> > > > 			  Parent.display(); // -> "Parent static method"
> > > > 			  Child.display(); // -> "Child static method"
> > > >         Parent p = new Child();
> > > >         p.display();  // -> "Parent static method"
> > > >     }
> > > > }
> > > > ```
> > 
> > > [!note]+ Dynamic Polymorphism
> > > > `Đa hình động` - hay còn gọi là `Đa hình khi chạy` hoặc `xác định muộn`.
> > > > Nó cho phép `một phương thức trong lớp con` có thể `ghi đè (override)` lại phương thức của `lớp cha` để thay đổi hành vi.
> > > 
> > > - **Cách thực hiện:**
> > >     - `Ghi đè phương thức` (`Method Overriding`): lớp con định nghĩa lại phương thức có cùng tên và tham số như trong lớp cha.
> > >     - Thường được triển khai thông qua `hàm ảo` (`virtual function`) trong C++ hoặc `kế thừa` trong Java.
> > > 
> > > ```java
> > > class Animal {
> > >     void sound() {
> > >         System.out.println("Animal makes a sound");
> > >     }
> > > }
> > > 
> > > class Dog extends Animal {
> > >     void sound() {
> > >         System.out.println("Dog barks");
> > >     }
> > > }
> > > 
> > > public class Main {
> > >     public static void main(String[] args) {
> > >         Animal a = new Dog(); // Đối tượng thực là Dog
> > >         a.sound(); // Kết quả: Dog barks
> > >     }
> > > }
> > > ```
> > > > [!note]+ Method Overriding
> > > > > `Method  Overriding` hay việc ghi đè phương thức là việc ta định nghĩa một method trong `Class con` (subclass) mà đã có sẵn trong `Class cha` (Parent). Như vậy, ta có thể định nghĩa được hành vi mở rộng cho class con mà không phải sửa cả class cha.
> > > > 
> > > > Một số quy tắc:
> > > > - Tên phương thức: phải trùng tên phương thức của Class cha
> > > > - Danh sách đối số: phải tương tự như Class cha
> > > > - Kiểu truy cập: `KHÔNG được bảo mật hơn phương thức của class cha`
> > > > ---
> > > > Ví dụ: Cha: `protected` → Con: **không thể** là `default` và `private`
> > > 
> > > > [!note]+ Covariant Return Type
> > > > > Khi bạn `Method Override` , các phương thức của class con sẽ có thể trả về một `kiểu con` của kiểu trả về trong phương thức gốc.
> > > > 
> > > > ```java
> > > > class Animal {
> > > >     Animal getAnimal() {
> > > >         System.out.println("Animal::getAnimal");
> > > >         return this;
> > > >     }
> > > > }
> > > > 
> > > > class Dog extends Animal {
> > > >     @Override
> > > >     Dog getAnimal() {  // Covariant return type
> > > >         System.out.println("Dog::getAnimal");
> > > >         return this;
> > > >     }
> > > > }
> > > > 
> > > > public class Test {
> > > >     public static void main(String[] args) {
> > > >         Animal a = new Dog();
> > > >         a.getAnimal();  // calls Dog::getAnimal
> > > >     }
> > > > }
> > > > ```
> > 
> > > [!note] 🗣
> > > `Compile-time`: dựa vào `kiểu tham chiếu` để overload, shadowing…
> > > `Run-time binding`: dựa vào `kiểu thực tế` để override
