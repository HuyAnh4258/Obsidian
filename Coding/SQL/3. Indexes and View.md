> [!note]+ # INDEXES
> > [!note]+ ### What is an index
> > > An index in database works like index in a book, it helps speed up the search but slow down the process of storing data
> > 
> > > [!note] 🗣
> > > When to use: 
> > > - If you have to retrieve data frequently from a table with ‘WHERE’ statement.
> > > - Data in a table is enormous (>2000 records), using INDEX is faster than normally scanning the whole table.
> > > 
> > > When not to:
> > > - If you have to use INSERT, UPDATE, DELETE  on a table frequently, using INDEX will slow down the insert process:
> > >     - It will have to find the index first before doing so.
> > >     - The indexes have to be updated every single time.
> > 
> > There are 2 types of indexes:
> > > [!note]+ Clustered
> > > - Data is stored in the order on the clustered index
> > > - Only 1 index per table (usually the PK)
> > > - ACTUALLY sort and store the data based on the key value
> > 
> > > [!note]+ Non-Clustered
> > > - Data is not sorted in the order of the index
> > > - DOES NOT sort data physically
> > > - Use pointers to point to the actual data
> > [!note]+ ### Creating an Index
> > - Create new index
> > 
> > ```sql
> > CREATE INDEX index_name
> > ON table_name (column1_name, column2_name, …)
> > ```
> > - Deleting an index
> > 
> > ```sql
> > DROP INDEX table_name.index_name
> > ```
> [!note]+ # VIEWS
> > [!note]+ ### What is a view?
> > > A View is a logical or virtual table. The fields in a view are fields from one or more real tables in the database.
> > View DOES NOT store actual data, just the equivalent query.
> > 
> > Why use view?
> > > [!note] 🗣
> > > 
> > > - Views allow you to limit the data users can access
> > > - Views reduce complexity for end users.
> > [!note]+ ### Create a view
> > ```sql
> > CREATE VIEW View_Name [list of column names]
> > AS
> > SELECT… --Select the data to be viewed here
> > ```
> > [!note]+ ### Delete a view
> > ```sql
> > DROP VIEW View_Name
> > ```
> [!note]+ # Naming Conventions
> 1. Use UPPER CASE for all T-SQL constructs, excepts Types
> ```sql
> --Correct:
> SELECT MAX([Salary]) FROM dbo.[EmployeeSalary]
> 
> --Incorrect:
> SELECT max([Salary]) from dbo.[EmployeeSalary]
> ```
> 2. Use lower case for all T-SQL Types and Usernames
> ```sql
> --Correct:
> DECLARE @MaxValue int
> 
> --Incorrect:
> DECLARE @MaxValue INT
> ```
> 3. Use Pascal casing for all UDO’s
> ```sql
> --Correct:
> CREATE TABLE dbo.EmployeeSalary
> (
> EmployeeSalaryID
> INT
> )
> 
> --Incorrect:
> CREATE TABLE dbo.Employeesalary
> (
> EmployeesalaryID
> int
> )
> ```
> 4. Avoid abbreviations and single character names
> ```sql
> --Correct:
> DECLARE @RecordCount int
> 
> --Incorrect:
> DECLARE @Rc int
> ```
> 5. UDO naming must confer to the following regular expression
>     - Do not use any special or language dependent characters to name objects. Constraints can use the underscore character.
> ```sql
> --Correct:
> CREATE TABLE dbo.[EmployeeSalary]
> 
> --Incorrect:
> CREATE TABLE dbo.[Employee Salary]
> ```
> 6. Use the following prefixes when naming objects
> | usp_: User stored procedures | DF_: Default constraints |
> | --- | --- |
> | ufn_: User defined functions | PK_: Primary Key constraints |
> | view_: Views | FK_: Foreign Key constraints |
> | IX_: Indexes | CHK_: Check constraints |
> |   | UNI_: Unique constraints |
> 
> ```sql
> --Correct:
> CREATE PROCEDURE dbo.usp_EmployeeSelectAll
> 
> --Incorrect:
> CREATE PROCEDURE dbo.EmployeeSelectRetired --without preffixed
> ```
> 7. Name tables in the singular form
> ```sql
> --Correct:
> CREATE TABLE dbo.[Employee]
> 
> --Incorrect:
> CREATE TABLE dbo.[Employees]
> ```
> 8. Tables that map one-to many, many-to-many relationships should be named by
> concatenating the names of the tables in question, starting with the most central table’s name.
> ```sql
> --Correct:
> CREATE TABLE dbo.[EmployeeSalary]
> ```
